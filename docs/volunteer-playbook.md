# Volunteer Playbook

## Where To Help

### Rust And Reliability

- config parsing
- status output
- approvals
- Windows and Linux launcher polish
- smoke tests and regression tests

### Documentation And Onboarding

- README improvements
- deployment walkthroughs
- contributor docs
- founder-brain reference indexing

### Deployment And Operations

- Docker hardening
- Linux host setup
- backup and restore guidance
- provider-switch testing

## First Contribution Path

1. Read the [project charter](project-charter.md).
2. Build the binary with `cargo build --release`.
3. Run `status` with the sample config.
4. Check [contributor-backlog.md](contributor-backlog.md) and pick a small issue tagged `good first issue`, `documentation`, or `help wanted`.
5. Open a PR with a short risk note and the commands you ran.

## Collaboration Norms

- Keep changes small and reviewable
- Preserve approvals and inspectable artifacts
- Ask whether a change fits the founder product shape before abstracting
- Prefer plain Markdown docs over new platforms

## Suggested Volunteer Roles

- Rust maintainer
- documentation maintainer
- deployment helper
- founder-brain curator
- QA and smoke-test helper

## Recognition

Maintainers should thank contributors in release notes, PR comments, and docs
where practical. Recognition matters more in a no-budget project because time,
trust, and momentum are the primary currency.
