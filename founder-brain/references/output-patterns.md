# Output Patterns

## Global Quality Rules

1. Deliver the artifact itself, not commentary about the artifact.
2. Do not wrap outputs in meta headings like `Draft`, `Output File`, `Verification Notes`, or `Ready for review` unless the request explicitly asks for that wrapper.
3. Do not use bracket placeholders such as `[Recipient Name]` in the main body. If a fact is missing, write `NEEDS_HUMAN_VERIFICATION` in the relevant spot or stop and say what is missing.
4. Do not claim proof points, demonstrations, partnerships, or project results unless they are grounded in the request or the approved references.
5. Do not use generic cold-email fillers such as `I hope this message finds you well`, `leading provider`, `successful demonstration`, or similar puffery unless quoted from the founder's own approved copy.
6. Keep outputs plain, reviewable, and close to the founder's actual voice: warm, scientific, direct, and anti-hype.
7. Do not add markdown emphasis, horizontal rules, or decorative formatting unless the request explicitly asks for markdown presentation.
8. Do not claim that something is verified unless the prompt packet or curated references actually verify it.

## Outreach Message Pattern

1. Open with a real observation about the recipient's work, ecology, or data challenge.
2. Name one narrow way you can help in 72 hours.
3. Mention one proof point or field example.
4. Offer a free micro-sample or quick review.
5. Close with a low-pressure next step.

## Offer Pattern

1. Name the audience and bottleneck.
2. Define the exact outcome.
3. List the inputs the client sends.
4. List what gets delivered within 72 hours.
5. State the micro-sample.
6. State the guarantee only if the scope is controllable.
7. Close with the smallest next action.

## Content Pattern

1. Start with a field image, number, or moment.
2. Explain what the data revealed.
3. Extract the deeper principle.
4. Tie it to MRV, biodiversity monitoring, or ecological infrastructure.
5. Close with one calm invitation.

## Pilot Pattern

1. State the site and question.
2. Name the data stack.
3. Define the indicators.
4. Describe the processing chain in plain language.
5. Define the 72-hour or first-phase deliverable.
6. Close with the next approval or dataset request.

## Decision Note Pattern

1. Frame the decision.
2. State the constraints.
3. Compare options briefly.
4. Recommend one path.
5. State the first action to take this week.

## External Draft Default

When asked for a client-facing email or outreach draft, default to:

1. `Subject A:`
2. `Subject B:`
3. blank line
4. the send-ready body
5. blank line
6. `Verification:` only if there are unresolved facts that the founder must confirm

## Internal Brief Default

When asked for an internal brief, note, or review:

1. Start with the decision or priority, not a motivational preamble.
2. Use short sections with concrete actions.
3. Name uncertainty plainly.
4. End with the next action, owner, or approval dependency.
